## 第二部分：Route Format 配置优化

### 当前配置（双字段）

```typescript
interface Route {
  request_format: 'openai' | 'anthropic' | 'gemini';
  response_format: 'openai' | 'anthropic' | 'gemini';
}
```

### 问题分析

**用户的问题**：是否只需要一个 format 字段？

**分析**：

| 场景 | request_format | response_format | 是否合理？ |
|------|----------------|-----------------|-----------|
| 直接透传 | openai | openai | ✅ 一致 |
| 格式转换 | openai | anthropic | ❌ 不一致？ |
| 格式转换 | anthropic | openai | ❌ 不一致？ |

### 关键洞察

**实际上，request_format 和 response_format 总是相同的！**

**原因**：
1. **Route 声明的是"对外提供的接口格式"**
2. **客户端既用这个格式发送，也期望用这个格式接收**
3. **如果需要转换，也是双向的（请求和响应都要转换）**

**结论**：✅ **可以简化为单一 `format` 字段**

### 优化后的配置

```typescript
interface Route {
  format: 'openai' | 'anthropic' | 'gemini';  // 🆕 单一字段
  // ... 其他字段
}
```

### 数据库迁移

```sql
-- 1. 添加新字段
ALTER TABLE routes ADD COLUMN format TEXT NOT NULL DEFAULT 'openai';

-- 2. 从 request_format 迁移数据
UPDATE routes SET format = request_format;

-- 3. 验证一致性（确保 request_format == response_format）
SELECT id, name, request_format, response_format
FROM routes
WHERE request_format != response_format;
-- （应该为空，如果不为空需要人工确认）

-- 4. 删除旧字段
ALTER TABLE routes DROP COLUMN request_format;
ALTER TABLE routes DROP COLUMN response_format;
```

### 代码修改影响

| 文件 | 修改内容 |
|------|---------|
| `src/shared/schema.ts` | Route 接口定义 |
| `src/server/module-gateway/services/route-matcher.service.ts` | 匹配逻辑 |
| `src/server/module-gateway/controllers/gateway-controller.ts` | 格式检测 |
| `scripts/seed-data.ts` | 种子数据 |

## 第四部分：关键发现总结

### 发现 1: Route Format 字段冗余

**问题**：`request_format` 和 `response_format` 总是相同

**原因**：
- Route 声明的是"对外接口格式"
- 客户端既用这个格式发送，也期望用这个格式接收
- 格式转换总是双向的

**建议**：✅ **简化为单一 `format` 字段**

### 发现 2: 当前系统都是直接透传

**现状**：
- 所有 Route 的 format 都匹配 Vendor 原生格式
- 无需格式转换
- 性能最优

**含义**：
- 当前架构是**正确且高效**的
- 格式转换功能存在但**未使用**

### 发现 3: 洋葱模型有 12 层

**层次分解**：
- 层 1-5: 请求处理（认证、匹配、验证、覆写、获取）
- 层 6: 格式检测
- 层 7A/10A: 格式转换（条件性，仅在需要时执行）
- 层 8: 日志创建
- 层 9: 上游通信
- 层 11: 客户端响应
- 层 12: 日志更新（异步）

**关键设计**：
- 格式转换是**条件分支**，不是必经之路
- 直接透传时完全跳过转换层
- 日志更新完全异步，不阻塞客户端

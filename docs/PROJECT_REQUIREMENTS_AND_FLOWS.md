## 1. 项目概述

### 1.1 项目定位

```
┌─────────────────────────────────────────────────────────────┐
│                     LLM Flux Gateway                        │
│                  统一的 LLM API 网关                        │
└─────────────────────────────────────────────────────────────┘

         │                    │                    │
         ▼                    ▼                    ▼
    ┌─────────┐         ┌─────────┐         ┌─────────┐
    │ OpenAI  │         │Anthropic│         │ Gemini  │
    │   API   │         │   API   │         │   API   │
    └─────────┘         └─────────┘         └─────────┘

         └──────────────────┬──────────────────┘
                            │
                    ┌───────▼────────┐
                    │  统一协议接口   │
                    │  (任意格式)     │
                    └────────────────┘
```

### 1.2 核心价值

- **协议统一**: 将多种 LLM API 格式统一为一种格式
- **智能路由**: 根据模型名称自动路由到最佳上游
- **灵活配置**: 支持参数重写、多资产负载均衡
- **完整监控**: 全链路日志追踪和性能分析
- **易于集成**: 兼容 OpenAI、Anthropic、Gemini 等主流格式

### 3.2 协议转换架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                    协议转换核心原理                                  │
└─────────────────────────────────────────────────────────────────────┘

  客户端请求              Internal Format           上游请求
  (任意格式)                  (统一)                (目标格式)

     │                          │                        │
     ▼                          │                        ▼
┌────────┐                      │                 ┌──────────┐
│ OpenAI │   ──转换───>         │                 │ OpenAI   │
│ Format │                      │                 │ Format   │
└────────┘                      │                 └──────────┘
     │                          │                        │
     ▼                          ▼                        ▼
┌──────────┐              ┌─────────┐            ┌──────────┐
│Anthropic │   ──转换───> │ Internal│  ──转换───> │Anthropic │
│ Format   │              │ Format  │            │ Format   │
└──────────┘              └─────────┘            └──────────┘
     │                          │                        │
     ▼                          ▼                        ▼
┌──────────┐              ┌─────────┐            ┌──────────┐
│ Gemini   │   ──转换───> │ Internal│  ──转换───> │ Gemini   │
│ Format   │              │ Format  │            │ Format   │
└──────────┘              └─────────┘            └──────────┘


核心设计原则:

1. Internal Format 使用 camelCase 命名
   promptTokens (而不是 prompt_tokens)
   completionTokens (而不是 completion_tokens)

2. 所有厂商格式差异在转换层处理
   OpenAI:    prompt_tokens  → promptTokens
   Anthropic: input_tokens   → promptTokens
   Gemini:    promptToken    → promptTokens

3. Gateway Controller 只访问 Internal Format
   不需要知道上游是什么格式
```

### 4.2 流式响应处理

```
┌─────────────────────────────────────────────────────────────────────┐
│                     流式响应处理流程                                 │
└─────────────────────────────────────────────────────────────────────┘

   Upstream SSE Stream                Client SSE Stream
        │                                     ▲
        │ 1. 接收 SSE chunk                    │
        ▼                                     │
   ┌─────────────┐                           │
   │ 解析 SSE     │                           │
   │ data: {...}  │                           │
   └──────┬──────┘                           │
          │ 2. 协议转换                        │
          ▼                                     │
   ┌─────────────────┐                        │
   │ Vendor Format   │                        │
   │ → Internal      │                        │
   └────────┬────────┘                        │
          │ 3. 提取增量内容                    │
          ▼                                     │
   ┌─────────────────┐                        │
   │ 提取 delta       │                        │
   │ 累积 tool_calls  │                        │
   └────────┬────────┘                        │
          │ 4. 协议转换                        │
          ▼                                     │
   ┌─────────────────┐                        │
   │ Internal Format │                        │
   │ → Client Format │                        │
   └────────┬────────┘                        │
          │ 5. 格式化 SSE                      │
          ▼                                     │
   ┌─────────────────┐                        │
   │ data: {...}\n\n │                        │
   └────────┬────────┘                        │
          │ 6. 发送给客户端                    │
          └──────────────────────────────────>┘
                     │
                     ▼
               ┌─────────┐
               │ 继续下个 │
               │ chunk   │
               └─────────┘


关键处理点:

1. 空 Chunk 处理
   - 上游可能发送只有 metadata 的 chunk
   - 检测: delta.content === "" && !delta.tool_calls
   - 动作: 跳过不发送

2. Tool Calls 累积
   - tool_calls 可能分多个 chunk 发送
   - 需要维护累积状态
   - 只在完整时才发送给客户端

3. 格式转换
   - OpenAI: choices[0].delta.content
   - Anthropic: delta.type === "content_block_delta"
   - Gemini: candidates[0].content.parts
```

```
     │                                    │<───────────────────────────┤
```

```
└─────────────────────────────────────────────────────────────────────┘

   输入: model="gpt-4-turbo", apiKeyId="key-123"
        │
        ▼
   ┌─────────────────────┐
   │ 查询所有路由         │
   │ WHERE isActive=true  │
   └──────────┬──────────┘
              │
              ▼
   ┌─────────────────────┐
   │ 过滤 API Key 权限    │
   │ 路由必须分配给该 Key │
   └──────────┬──────────┘
              │
              ▼
   ┌─────────────────────┐
   │ 评估匹配度           │
   │ - 精确匹配: 100 分   │
   │ - 通配符: 50 分      │
   │ - 未匹配: 0 分       │
   └──────────┬──────────┘
              │
              ▼
   ┌─────────────────────┐
   │ 按优先级排序         │
   │ 1. 匹配度降序        │
   │ 2. priority 降序     │
   │ 3. 创建时间升序      │
   └──────────┬──────────┘
              │
              ▼
   ┌─────────────────────┐
   │ 返回最佳匹配         │
   │ 第一条路由           │
   └─────────────────────┘


示例场景:

路由配置:
┌──────────────┬────────────────┬───────────┬────────────┐
│ 路由名称      │ 模型匹配规则    │ 优先级    │ API Key    │
├──────────────┼────────────────┼───────────┼────────────┤
│ GPT-4 路由    │ gpt-4-*        │ 100       │ key-123    │
│ Claude 路由   │ claude-*       │ 90        │ key-123    │
│ 默认路由      │ *              │ 10        │ key-123    │
└──────────────┴────────────────┴───────────┴────────────┘

请求 1: model="gpt-4-turbo"
         匹配: GPT-4 路由 (通配符) + 默认路由 (通配符)
         结果: GPT-4 路由 (优先级更高)

请求 2: model="claude-3-opus"
         匹配: Claude 路由 (通配符) + 默认路由 (通配符)
         结果: Claude 路由 (优先级更高)

请求 3: model="unknown-model"
         匹配: 默认路由 (通配符)
         结果: 默认路由
```

## 附录

### A. 术语表

| 术语 | 定义 |
|------|------|
| Internal Format | 统一的中间格式，使用 camelCase 字段命名 |
| Vendor Format | 各厂商的 API 格式（OpenAI/Anthropic/Gemini） |
| Converter | 协议转换器，负责在不同格式间转换 |
| Route | 路由配置，定义请求如何转发到上游 |
| Asset | 上游资产，包含厂商 API 凭证 |
| Override | 重写规则，修改请求参数 |
| SSE | Server-Sent Events，流式响应格式 |
| TTFB | Time To First Byte，首字节时间 |

| `/api/keys` | GET/POST | API Key 管理 |
| `/api/routes` | GET/POST/PUT/DELETE | 路由配置管理 |

| `/api/logs` | GET | 请求日志查询 |

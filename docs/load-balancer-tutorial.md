# 负载均衡功能使用教程

## 🎯 功能入口

负载均衡功能在 **Access Keys** 页面中管理。

**访问路径：**
1. 登录系统
2. 点击左侧菜单的 **🔑 Access Keys**

---

## 📋 功能概述

### 负载均衡能做：
- ✅ 一个 API Key 绑定多个路由
- ✅ 按权重比例分配请求
- ✅ 健康状态自动检测
- ✅ 失败自动切换到下一个路由

---

## 🚀 快速开始

### 步骤 1：创建路由

在配置负载均衡之前，先创建多个路由：

1. 进入 **Route Flux** 页面
2. 创建 2-3 个不同的路由（例如：不同的 AI 服务商）
3. 记录路由名称

### 步骤 2：创建 Access Key 并配置路由

1. 进入 **Access Keys** 页面
2. 点击 **Create Key** 或选择一个已有的 Key
3. 点击 **Edit Routes** 按钮

### 步骤 3：配置权重

在编辑路由对话框中：

```
路由列表：
☑ Route 1 (GPT-4)     权重: 100  🟢 健康
☑ Route 2 (Claude)     权重: 50   🟢 健康
☑ Route 3 (Gemini)     权重: 25   🟡 降级
```

**权重配置说明：**
- 权重越高，被选中的概率越大
- 权重为 0 的路由不会被使用
- 总权重不影响比例，只影响相对分配

**示例：**
```
Route 1: weight = 300  → 被选中 75% 的时间
Route 2: weight = 100  → 被选中 25% 的时间
```

### 步骤 4：保存配置

点击 **Save** 保存配置

## ⚙️ 健康状态说明

### 状态类型

| 状态 | 颜色 | 说明 | 是否使用 |
|------|------|------|----------|
| **Healthy** | 🟢 绿色 | 正常工作 | ✅ 使用 |
| **Degraded** | 🟡 黄色 | 部分失败 | ⚠️ 降低权重使用 |
| **Unhealthy** | 🔴 红色 | 持续失败 | ❌ 跳过 |

### 自动切换逻辑

当请求失败时：
1. 失败计数 +1
2. 如果失败次数 ≥ 3，标记为 Unhealthy
3. 跳过不健康的路由
4. 选择下一个健康的路由

当请求成功时：
1. 成功计数 +1
2. 如果成功率 ≥ 80%，标记为 Healthy
3. 恢复正常使用

- 健康状态

## 🎨 高级配置

### 权重调整策略

**场景 1：灰度发布**
```
新版本 Route: weight = 10   (10% 流量)
旧版本 Route: weight = 90  (90% 流量)
```

**场景 2：AB测试**
```
Route A: weight = 50  (50% 流量)
Route B: weight = 50  (50% 流量)
```

**场景 3：主备切换**
```
主 Route: weight = 100  (全部流量)
备 Route: weight = 0    (不启用)
```

### 故障恢复

当主路由恢复后：
1. 自动将权重调回 100
2. 备路由权重调回 0

## 🔧 常见问题

### Q1: 为什么某些路由没被选中？

检查：
1. 权重是否为 0？
2. 健康状态是否为 Unhealthy？
3. 路由是否启用（is_active）？

### Q2: 权重配置后多久生效？

立即生效。每次请求都会使用最新的权重配置。

### Q3: 如何查看路由选择分布？

在 **Logs** 页面查看多次请求的 `route_id` 分布。

### Q4: 可以动态调整权重吗？

可以。在 **Access Keys** 页面编辑路由权重，实时生效。

## 📝 API 接口

### 更新路由权重

```bash
PUT /api/keys/:id/routes
Content-Type: application/json
Authorization: Bearer <token>

{
  "routeWeights": [
    { "routeId": "route-1", "weight": 300 },
    { "routeId": "route-2", "weight": 100 }
  ]
}
```

### 获取 Key 详情（包含健康状态）

```bash
GET /api/keys/:id
Authorization: Bearer <token>

Response:
{
  "id": "key-001",
  "name": "Production Key",

      "weight": 100,
      "healthStatus": "healthy",
      "failCount": 0,
      "successCount": 1000,
      "avgLatencyMs": 150

  ]
}
```

   - 监控平均延迟

   - 定期健康检查

## 🆘 故障排查

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| 所有路由都不健康 | 网络问题 | 检查网络连接 |
| 权重不生效 | 配置未保存 | 重新保存 |
| 请求一直失败 | API Key 问题 | 检查上游配置 |
| 延迟很高 | 路由过载 | 降低权重 |
